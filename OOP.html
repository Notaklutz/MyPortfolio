Object-Oriented Programming (OOP)


Java is an object-oriented programming (OOP) language. As such, programming in Java revolves around using "objects". Simply put, an object is an instance of a class. A good way to visualize this concept is to think of a class as a template and an object as a creation using that template. Each object of a class has certain attributes (variables) and behaviours (methods). Every object can have different specifications for their variables, but they still retain the same attributes. In this way, OOP mirrors real life. For example, every person has certain attributes (eye colour, hair colour, height, etc.) and behaviours (walk, sleep, talk, etc.). However, each person has different eye colours, heights, etc. Some people might even walk or talk differently (we can mimic this by overriding methods, which will be explained in a different section). We can use a template (class) to create a generic person (object) and customize its attributes (variables). Throughout this section, we will continue to use the Person class as an example.

 

Classes

A class defines an object and contains the methods that will interact with it. Every class (with the exception of a few, which will be explained later) has a constructor. If the programmer does not define one, Java will automatically create one with no parameters. A constructor is used to create instances/objects of a class. A class can have many constructors with different parameters to create objects with different specified attributes. For example, one constructor might make a person with blue eyes and another might make a person with brown eyes. Constructors usually initialize all of an object's instance variables. They also don't have a return type or return a value and have the same name of the class. Another important concept of classes in Java are static methods. These methods can be used without any objects of the class being created. For example, the random() method in the Math class does not need a Math object to be used. For non-static methods, an object must be created for the method to be useable.

The Person class has two constructors. One will initialize the instance variables with default values and one will initialize the instance variables according to the parameters that are passed into the constructor.

 

Objects

As stated before, objects are instances of a class. They contain data (instance variables/fields) and methods that operate on the aforementioned data. Objects are created using the keyword new followed by the class name.


The new keyword is used to create a Person object with black eyes, red hair, and a height of 160.

 

Parent Classes (Superclasses), Subclasses, and Inheritance

Classes can be extended and made subclasses of other classes. This means that they inherit all of the methods and fields of their parent class. For example, the Child class will extend the Person class. It inherits the fields (height, eye colour, etc.) and methods (walk, talk, etc.) of the Person class. When a subclass extends a parent class (also known as a superclass), it has a IS-A relationship with that class. For example, a child IS-A person. Furthermore, each subclass can even override methods of its parent class by redefining it within the subclass. For example, an olympic sprinter will not have the same run() method as a toddler. Technically, all classes are subclasses, as all classes implicitly extend the Object class.

The Child class extends Person. Person is now the parent class of Child, which is now a subclass of Person. Notice that the Child constructor uses the same instance variables as Person, despite not declaring any instance variables. This is because Child inherits the instance variables of Person, its parent class.

 

Encapsulation (Public and Private)

Constructors and methods that are declared public can be accessed by other classes. Usually, all instance variables and helper methods (methods that are only needed inside the class) are declared private and hidden from other classes (they can only be accessed by the class they're in). The process of declaring instance variables and helper methods as private is known as encapsulation. It prevents classes from being changed if an external class is changed and is crucial in team development as classes can only be interacted with via its public constructors and methods. In order to access private fields, accessor (getter) methods and mutator (setter) methods are used.


All of the instance variables in Person are declared private. They can only be interacted with by accessor and mutator methods. In the case of eyeColour, these methods are getEyeColour() and setEyeColour(). Notice that setEyeColour() allows for a parameter pass in order to set the eye colour.

 

Abstraction

Abstraction is the process of creating a generic class (containing shared methods and fields) that will be used as the parent class of many subclasses. For example, the Person class contains shared methods and fields and will be used as the parent class of the Adult class, Child class, Teenager class, etc.

 

Polymorphism

Polymorphism occurs when an object extends or implements another class and is able to be represented as something other than itself. The easiest way to visualize this is to check if a class satisfies more than one IS-A relationship. In this way - technically speaking - all classes are polymorphic as all classes implicitly extend the Object class. For example, the SeniorCitizen class is polymorphic. It IS-A Object, IS-A Person, and IS-A SeniorCitizen.


The SeniorCitizen class explicitly extends Person and implicitly extends Object. It is polymorphic because it IS-A SeniorCitizen, IS-A Person, and IS-A Object.

 

Upcasting and Downcasting

Upcasting is treating an object of a subclass as an object of a superclass and downcasting is treating an object of a superclass as an object of a subclass. For example, an object of the Swimmer class can be upcast to the Athlete class. This allows the Swimmer object to use methods of the Athlete class. However, the Swimmer object itself is not changed, it is just being represented differently. While being represented as an Athlete however, the Swimmer object cannot use methods of the Swimmer class (swim() for example). Unlike downcasting, upcasting is done automatically. This is because upcasting can never fail. For example, a Swimmer object and a HockeyPlayer object can always be upcast to Athlete, but a Swimmer object, while being represented as an Athlete, cannot be downcast to HockeyPlayer without an error. This is why downcasting must be done manually by the programmer. Upcasting and downcasting are useful because they allow for the usage of general methods in superclasses, which simplifies subclasses. For example, both Swimmer and HockeyPlayer can be automatically upcast to Athlete to use its run() method.


The Swimmer object sally and the HockeyPlayer object horton are being automatically upcast to Athlete in order to use the Athlete class' run() method.
The Swimmer object is automatically upcast to Athlete but needs to be manually downcast back to Swimmer.

 

Special Classes

Final classes, abstract classes, and interfaces are all special types of classes. A final class cannot be extend another class or have itself be extended. Furthermore, none of its methods can be changed. These classes are very secure owing to the fact that they can never be modified. The String class is an example of a built-in final class. Abstract classes are extremely basic classes that act as the ultimate template for any subclass. They are meant to be extended by another class and must contain at least one abstract method that will be overriden by its subclass. Although abstract methods must be declared, they do not need to have any actual implementation. Instead, it is the job of the subclass to define an abstract method. For example, the Person class could be an abstract class that contains an abstract method called talk(). Each subclass would then be able to define talk() differently, as everyone doesn't talk the same (for example, an English person vs a French person). Finally, an interface is a class that contains only public abstract methods and (potentially) constants (public static final variables). Since all methods in an interface are abstract, the public and abstract keywords do not have to be explicitly stated in the method declarations. Interfaces are meant to be implemented and any class that implements them must define all of their methods. Interfaces are used to define behaviours that are common among different objects. For example, an interface called JumpCapable (able to jump) might be implemented by the Adult class, Teenager class, maybe even some Animal subclasses, but not by the Baby class. Interfaces can be implemented regardless of what a class extends (in the previous example, this was demonstrated by a subclass of Animal implementing the JumpCapable interface). Any class that implements an interface can also be considered polymorphic (an Adult IS JumpCapable;see polymorphism above for in-depth explanation). Unlike an abstract class however, implementations cannot be written for any methods of an interface and an interface cannot contain instance variables (only constants). Both abstract classes and interfaces cannot be instantiated. Their only purpose is to be extended (abstract classes) or implemented (interfaces).


Person is an abstract class that contains the abstract method run().
JumpCapable is an interface that contains a constant (gravity) and a public abstract method called jump(). Notice that Java automatically defines every method in an interface to be public abstract, thereby removing the need for the programmer to define each method as such.
