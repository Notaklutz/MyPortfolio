Object-Oriented Programming (OOP) Examples


Don't Fear the OOP

Don't Fear the OOP is a Java tutorial that was created by Johannes Claerbout in 1998. It uses the process of a writer creating a Western novel to explain the concept of OOP in Java. In my AP Computer Science class, I, along with two others, adapted this concept to explain OOP using George Lucas' Star Wars movies. This assignment allowed me to gain a thorough understanding of how to design objects and classes in Java in a fun and easy-to-understand way. Creating UML Class Diagrams

Being able to create UML (Unified Modelling Language) class diagrams is an important part of class and program design. Not only can they give you can overview of an individual class, but they can also clarify the relationships that exist between multiple classes in a program. In the examples on the side, I've made a class diagram for a single class, as well as a larger diagram showing the relationships between multiple classes.


In this individual class diagram, there are two sections. The top section is for the instance variables (including data types) while the bottom section is for the methods and constructors (including parameters and return type). Before each variable/method/constructor, there is a symbol denoting its access modifier. - is for private, + is for public, and # is for protected.

In this UML diagram, multiple classes and their relationships with other classes are displayed. Each class has an individual class diagram, showing their instance variables/methods/constructors. The Rectangle, Triangle, and Oval classes have an arrow connecting them to the Shape class, which shows that those classes extend Shape and have an is-a relationship with it.

 

Abstract Shapes

In the following example, I demonstrate important OOP concepts such as abstract classes and methods, parameter passing, inheritance, and encapsulation by having two classes (RectangularPrism and Sphere) extend an abstract class (Solid). The subclasses inherit the instance variables and methods of Solid. They also implement the abstract methods, which have no implementation in Solid. Furthermore, all of the instance variables are private, which means that they can only be accessed and manipulated through methods such as getName(), volume(), and surfaceArea().
